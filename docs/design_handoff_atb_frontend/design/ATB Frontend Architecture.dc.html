02 — What is driving these choices

The app decides the architecture

ATB is not a content site. It is a dense, authenticated, real-time operations console: engineers watch agent squads stream work, approve diffs, and steer running sessions. Four properties of the product dominate every decision below.

It is real-time first, not real-time later
Sessions stream agent tokens continuously; status, cost, and squad phase change live. Streaming is the spine — SSE for one-way token feeds, WebSockets for bidirectional control — and the state layer is chosen to absorb those events cleanly.
It is broad — a dozen first-class surfaces
Overview, Sessions, Runs, Workflows, Agents, Squads, Skills, MCP servers, Sandboxes, Analytics — each with list and detail views. Breadth demands a feature-sliced structure a solo dev (and Claude Code) can navigate without cross-cutting spaghetti.
It is built solo, with an AI pair
The stack favors strong conventions and legible file layout over cleverness — one obvious place for everything, generated types instead of hand-written glue, and a framework whose patterns Claude Code already knows deeply. Convention is the force multiplier.
It ships to enterprise AWS ECS
No serverless assumptions. Everything must run as a long-lived container behind an ALB — which shapes the streaming, session-affinity, and horizontal-scaling design more than any other constraint. It also keeps the door open to air-gapped / on-prem later.
03 — The stack, layer by layer

Every pick, the reason, the runner-up

Each layer names one primary choice and the alternative you would reach for under a different constraint. Where a popular option was deliberately declined, that is stated too.

L1 Framework & Runtime

Next.js 16

React 19.2 · App Router

The App Router gives us React Server Components — most of a dozen data-heavy surfaces render on the server and ship near-zero JS, which is exactly what a broad console needs for fast first paint. Turbopack is the default bundler in 16 (fast HMR on a large app), the React Compiler removes hand-written memoization, and Route Handlers give us the BFF in the same repo — including streaming responses for SSE and a WebSocket upgrade path. It runs as a standalone Node container on ECS with no serverless lock-in, and its conventions are the ones Claude Code reasons about most reliably.

Alternative
Vite + React Router 7 (framework mode) with a standalone Hono BFF — a stricter SPA/API split if you'd rather deploy the UI and API as separate services.
Declined
TanStack Start — excellent, but younger; the ecosystem depth and hiring/AI familiarity of Next tips it for a system meant to last.
L2 Language & Contracts

TypeScript 5.9 + Zod 4

strict: true

Strict TypeScript everywhere — no implicit any, no unchecked boundaries. Zod is the single source of truth for shapes: one schema validates an inbound agent event at the BFF, types the tRPC output, and powers form validation on the client. A parse failure is a caught error, never a silent undefined three renders later — critical when the data arrives as a live stream you don't fully control.

Alternative
Valibot where bundle size is tight (it tree-shakes smaller) — the schemas read almost identically.
Note
Generate types from the backend's OpenAPI/proto once it exists; keep Zod for runtime-validated boundaries (streams, forms, webhooks).
L3 API Boundary — the BFF

Route Handlers + tRPC v11

Since there is no backend yet, the frontend ships with its own Backend-for-Frontend: a thin server layer that owns auth/session, holds secrets, and shapes data for the UI. tRPC gives fully typed, autocompleted calls with zero code-gen and zero drift between server and client — the biggest single productivity lever for a solo TS developer. When real backend services (the agent orchestration engine, Postgres, the MCP registry) come online, the BFF becomes the aggregation seam that calls them; the UI contract never changes.

Alternative
oRPC when the contract must be OpenAPI-documented for external/mobile consumers — same ergonomics, spec-first output.
Declined
GraphQL — the client-flexibility win doesn't pay for its operational weight with a single first-party UI and a BFF you fully control.
L4 State — server, client, URL

TanStack Query · Zustand · nuqs

The trap in real-time apps is dumping everything into one global store. We split state by ownership. TanStack Query owns all server data — caching, deduping, background refetch — and is the single sink that streaming events reconcile into. Zustand owns the small amount of true global UI state (command palette, panel layout, ephemeral toasts). nuqs puts session filters, the selected session, and the active canvas tab in the URL — so every view is shareable, reloadable, and back-button-correct.

Alternative
Jotai for atomic state if fine-grained derived subscriptions become the ergonomic bottleneck.
Declined
Redux Toolkit — the boilerplate and single-store model fight against server-owned, stream-driven data.
L5 Real-time transport ★ the crux

SSE for streams · WebSocket for control

Use the right pipe for each job rather than forcing one. Server-Sent Events carry the high-frequency, one-way flows — agent token output, run logs, live status ticks. SSE auto-reconnects, survives proxies, and is dramatically simpler to operate than a socket. WebSockets carry the genuinely bidirectional traffic — steering a running session, approvals, presence, interrupts. Both feed a single reconcile layer that writes into the Query cache via queryClient.setQueryData, so the UI never learns which transport a change arrived on. Full flow in §05.

SSE — fetch-event-source
POST-capable, header-aware streaming reader (not raw EventSource, which can't send auth headers). Token feeds, logs, status.
WS — partysocket client
Transport-agnostic reconnecting WebSocket with backoff & heartbeat. Steering, approvals, presence, interrupts.
Alternative
Collapse both onto one WebSocket only if operational simplicity outweighs SSE's reconnect/proxy resilience for your infra.
ECS caveat
Across multiple tasks a client's socket lands on one task only — a Redis pub/sub backplane fans events to all. See §06.
L6 Styling & Theming

Tailwind CSS v4 + CSS variables

The prototype already is a semantic token system — --bg, --text, --accent, dark/light swapped on [data-theme]. Tailwind v4's CSS-first @theme binds directly onto those exact variables, so theming stays in CSS (where the runtime toggle lives) while utilities give velocity. Keep the tokens as the contract; Tailwind is just ergonomic access to them.

Alternative
vanilla-extract — typed, zero-runtime CSS if you prefer authored stylesheets with TS safety over utility classes.
Declined
Runtime CSS-in-JS (styled-components/Emotion) — per-render style cost and RSC friction make it the wrong call here.
L7 Component primitives

shadcn/ui + Radix + cmdk

shadcn/ui copies component source into the repo rather than importing a black box — you own and restyle it to the ATB tokens, and Claude Code can read and edit it like any other file. It sits on Radix primitives (accessible dialogs, popovers, dropdowns, tooltips with focus management built in). cmdk powers the ⌘K command menu; Sonner handles toasts. These are floors, not ceilings — bespoke surfaces (the session canvas) are built directly.

Alternative
Base UI (the Radix/Floating-UI/MUI team's 1.0 successor) — the forward path as shadcn adopts it.
Declined
MUI / Ant Design — heavy opinionated themes fight the crafted ATB look; you'd spend more overriding than building.
L8 Specialized libraries

The right tool per surface

Focused dependencies for the surfaces that need them — each small, tree-shakeable, and replaceable.

TanStack Table + Virtual
Runs & sessions are long, live-updating lists — virtualization keeps them at 60fps. Alt: AG Grid for spreadsheet-grade needs.
Recharts
Analytics cost / model-mix charts. Alt: visx for bespoke viz, Tremor for instant dashboards.
Shiki

05 — Real-time data flow

How a token reaches the screen

The defining pattern of the whole app: streaming transports never touch React state directly. They land in one reconcile function that updates the Query cache; components just render the cache. This keeps optimistic control, refetch, and live streams from fighting each other.

1
Agent engine emits — tokens, tool calls, status deltas as an event stream.
↓
2
BFF proxies over SSE — attaches auth, multiplexes per session, publishes to Redis so every task can serve it.
↓
3
Client SSE reader parses + Zod-validates — each frame becomes a typed event or a caught error.
↓
4
reconcile() writes the cache — queryClient.setQueryData(['session', id], append). One function, one sink.
↓
5
Components re-render from cache — the transcript grows; no component knows SSE exists.
The return path (WebSocket)
User clicks Approve or Interrupt → optimistic cache update → WS message to the BFF → engine acts → the authoritative echo arrives back through the same reconcile function, confirming or rolling back. Bidirectional control and one-way streams share a sink, never a transport.
06 — Deployment on AWS ECS

Containers, streaming, and the fan-out problem

Next.js builds to a standalone Node server in a small Docker image and runs as a long-lived Fargate service — no serverless caveats around streaming or connection lifetime. The one real design decision is horizontal scaling of live connections.

Ship path
GitHub Actions builds & tests → Docker image → ECR → ECS Fargate rolling / blue-green via CodeDeploy → behind an ALB. Static assets to S3 + CloudFront.
Config & infra
Secrets in Secrets Manager, config in SSM Parameter Store. All infrastructure as AWS CDK (TypeScript) — same language as the app.
The critical production detail — WebSocket fan-out

With more than one Fargate task, a client's WebSocket is pinned to one task. An event produced on task A must reach a subscriber on task B. Solve it with a Redis (ElastiCache) pub/sub backplane: every task publishes/subscribes to session channels, so any task can serve any client. Configure the ALB with a long idle timeout for SSE/WS, disable response buffering on stream routes, and enable stickiness only as a fallback. Design this in from Phase 1 — retrofitting a backplane into a live socket layer is painful.

07 — Folder & module structure

Feature-sliced, one obvious home per thing

app/ stays thin — routing only. Each domain lives in a self-contained features/<name>/ slice with its own components, hooks, schema, and store. This is the layout a solo dev and Claude Code navigate fastest: to change Sessions, everything is in one folder.

08 — Phased implementation roadmap

Spine first, surfaces second

Roughly twelve weeks solo with Claude Code. Build the real-time spine and app shell before any feature surface — every surface depends on them, so proving them early de-risks everything after. Indicative durations, not commitments.

PHASE 0Week 1

Foundation

Repo, pnpm, strict TS, Biome, lefthook. Tailwind v4 wired to the ATB token file. shadcn initialized. CI skeleton, Dockerfile, CDK bootstrap, and the MSW mock harness — so day-one work runs against fake streams.

PHASE 1Weeks 2–3 · the spine

Shell & real-time spine

App shell: nav, breadcrumb, workspace/env switcher, ⌘K, dark/light theming. Auth at the BFF. tRPC + Query wired. The SSE + WS clients and the reconcile() layer built and load-tested with the Redis backplane. Deploy to an ECS staging service.

PHASE 2Weeks 4–6 · hero surface

Sessions

The grouped, virtualized sessions list and the session detail — streaming chat plus the canvas (plan / run / diff / trace) with Shiki-highlighted diffs. This is where the spine pays off; ship it end-to-end before widening.

PHASE 3Weeks 7–9

Build surfaces

Agents (model, edit/bash/network permissions, system prompt), Squads, Workflows, Skills, and MCP servers with health states. Mostly forms + detail views reusing the slice pattern — fast once the conventions are set.

PHASE 4Weeks 10–11

Runtime & insight

Runs history + failures, Sandboxes, the Analytics dashboard (Recharts: cost, model mix), and the Overview home that ties the activity feed together.

PHASE 5Week 12+ · ongoing

Hardening & cutover

Accessibility pass, performance-budget enforcement in CI, visual-regression coverage, WS fan-out load test, observability dashboards, and the blue-green production cutover.
